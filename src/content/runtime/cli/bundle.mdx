## Synopsis

```bash
phrony bundles ls [flags]
phrony bundles lock BUNDLE
phrony bundles validate BUNDLE [--require-lock]
phrony bundles publish BUNDLE
phrony bundles deploy BUNDLE@VERSION
phrony bundles versions BUNDLE
phrony bundles active BUNDLE
phrony bundles history BUNDLE
phrony bundles run BUNDLE[@VERSION] [flags]
phrony bundles secret-requirements BUNDLE[@VERSION]
```

## Description

Multi-agent systems that use [`spec.agents`](/docs/agent-spec/resources/agents) ship as [`kind: Bundle`](/docs/agent-spec/resources/bundle). The `bundles` command group walks the delegation closure, commits a **`bundle.lock.json`** sidecar, and publishes an immutable bundle version keyed by **`metadata.version`** (semver) and the lock hash (`sha256:â€¦`).

Only [`bundles lock`](/docs/runtime/cli/bundle-lock) writes the lock file. [`bundles publish`](/docs/runtime/cli/bundle-publish) requires a committed lock and verifies it before contacting the runtime.

## Typical workflow

```bash
# Edit closure members, bump metadata.version in bundle.yaml, then:
phrony bundles lock ./support/bundle.yaml            # add --runtime-addr when closure has external refs
phrony bundles validate ./support/bundle.yaml --require-lock
phrony bundles publish ./support/bundle.yaml         # â†’ support/support 1.0.1 (sha256:â€¦)
phrony bundles deploy support.support@1.0.1
phrony bundles run support.support
```
